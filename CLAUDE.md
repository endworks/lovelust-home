## Architecture

This is a **Next.js 16 / React 19** app ‚Äî a marketing/landing site for the LoveLust mobile app.

### Rendering model

Pages are split into a **server component** (reads files, exports `metadata`) and a **`*Client.tsx` component** (marked `"use client"`, handles all interactivity). The pattern used throughout:

- `app/page.tsx` ‚Üí reads markdown from `content/`, exports metadata, renders `<HomeClient>`
- `app/privacy/page.tsx`, `app/terms/page.tsx`, `app/support/page.tsx` ‚Äî same pattern

### i18n

Two languages: **English** (`en`) and **Spanish** (`es`). Translation strings live in `public/locales/{en,es}/common.json`. The i18next instance (`src/i18n.ts`) is initialized with `initAsync: false` (synchronous) so the correct language is active before React's first render. Language is detected from `navigator` on the client. The server always SSRs in `"en"`.

Long-form content (FAQ, Privacy Policy, Terms) is authored as Markdown in `content/{faq,privacy,terms}/{en,es}.md` and rendered via `react-markdown`.

### Styling

**No Tailwind / CSS framework** ‚Äî all layout is inline `style={{}}` with a few utility classes. Design tokens are CSS custom properties defined in the `:root` of `app/globals.css`:

- `--accent: #f61e6d` (pink/fuchsia ‚Äî primary brand color; `--accent-dark`, plus alpha variants `--accent-06/-0d/-12/-15/-18`)
- `--secondary: #73575c` (muted mauve; `--secondary-08`, `--secondary-bg`)
- `--bg`, `--text`, `--text-muted`, `--border` ‚Äî switch between light/dark via `.dark` class on `<html>`

Dark mode is system-preference-only (no manual toggle persisted). An inline `<script>` in `<head>` reads `prefers-color-scheme` and adds the `dark` class before React hydrates.

### Components

- `components/sections/` ‚Äî page sections (Hero, Features, AppPreview, Values, Testimonials, FAQ, CTA)
- `components/` ‚Äî shared UI: `Header`, `Footer`, `StoreBadge`, `StoreButton`, `PhoneMockup`, `RatingStars`, `FaqList`, `MarkdownContent`, `PageLayout`, `ThemeAwareScreenshot`, `SectionContainer`
- `hooks/` ‚Äî `useTheme`, `useMediaQuery`, `usePageTracking`

### Analytics

Aptabase (`@aptabase/react`) wraps the app in `app/providers.tsx`. Page tracking is called via `usePageTracking(pageName)` inside each `*Client.tsx`.

### Build mode

`next.config.ts` always builds `output: "standalone"` (Docker/self-hosted Node
server). This is required: the campaign QR short URLs (`/card-26`, `/erospain-26`,
‚Ä¶) are server-side 301 redirects. A static export would silently drop them and
404 every already-printed QR code. Do not reintroduce a static-export build.

### SEO files

All generated by Next.js conventions:

- `app/robots.ts` ‚Üí `/robots.txt`
- `app/sitemap.ts` ‚Üí `/sitemap.xml`
- `app/manifest.ts` ‚Üí `/manifest.webmanifest`
- JSON-LD structured data (`WebSite`, `Organization`, `MobileApplication`, `FAQPage`) is inlined in `app/page.tsx`

### Environment variables

All prefixed `NEXT_PUBLIC_`. Key ones:

- `NEXT_PUBLIC_APP_URL` ‚Äî canonical site URL (falls back to `https://lovelust.health`)
- `NEXT_PUBLIC_APPSTORE_URL` / `NEXT_PUBLIC_GOOGLE_PLAY_STORE_URL` ‚Äî store links
- `NEXT_PUBLIC_APTABASE_APP_ID` ‚Äî analytics key
- `NEXT_PUBLIC_ASA_PROVIDER_TOKEN` ‚Äî optional Apple `pt` provider token added
  to App Store links for install attribution (campaign `ct` is derived from UTM)
- Hero rating (`lib/storeRatings.ts`): App Store rating is auto-fetched from
  Apple's iTunes Lookup API (id parsed from `NEXT_PUBLIC_APPSTORE_URL`, or
  `NEXT_PUBLIC_APPSTORE_ID`), cached 24h. Google Play has no official API:
  `NEXT_PUBLIC_PLAY_RATING` / `NEXT_PUBLIC_PLAY_RATING_COUNT` ‚Äî keep in sync
  with the Play Console. Missing/failed values fall back to a plain chip.

### Testing

Unit tests use **Vitest + jsdom + Testing Library**. Setup file at `tests/setup.ts` mocks `matchMedia`, `next/navigation`, and `react-i18next`. E2e tests use Playwright in `e2e/`.
